# Plan 07-02: SharedArrayBuffer Bridge and COOP/COEP Headers

## Objective

Enable SharedArrayBuffer support for zero-copy grid data sharing between the Web Worker (where WASM runs) and the main thread (where rendering reads buffers). Add required COOP/COEP headers to Next.js config. Make Grid optionally use SharedArrayBuffer-backed Float32Arrays so that the Worker's WASM writes are immediately visible to the main thread's renderer without postMessage transfers.

## must_haves

- Next.js config adds Cross-Origin-Opener-Policy and Cross-Origin-Embedder-Policy headers
- Grid can optionally use SharedArrayBuffer-backed Float32Arrays
- Feature detection: if SharedArrayBuffer is unavailable, fall back to regular ArrayBuffer silently
- Grid.getCurrentBuffer() returns the same typed array view regardless of backing store type
- No breaking changes to existing Grid API -- all existing tests continue to pass
- Unit tests for SharedArrayBuffer-backed grid creation and buffer sharing

## Tasks

<task id="1">
Update `next.config.ts` to add COOP/COEP headers required for SharedArrayBuffer:
- Add `headers()` async function returning custom headers for all routes
- Headers: `Cross-Origin-Opener-Policy: same-origin`, `Cross-Origin-Embedder-Policy: require-corp`
- Keep existing webpack config for asyncWebAssembly
</task>

<task id="2">
Update `src/engine/grid/types.ts`:
- Add `useSharedBuffer?: boolean` optional field to `GridConfig`
- Add `SharedPropertyBuffers` type that extends `PropertyBuffers` with shared backing info
</task>

<task id="3">
Update `src/engine/grid/Grid.ts` to support SharedArrayBuffer backing:
- In `addProperty()`: if `config.useSharedBuffer` is true and `typeof SharedArrayBuffer !== 'undefined'`, create buffers backed by SharedArrayBuffer:
  ```
  const sab = new SharedArrayBuffer(bufferSize * Float32Array.BYTES_PER_ELEMENT);
  const buffer = new Float32Array(sab);
  ```
- If SharedArrayBuffer is not available (feature detection), fall back to regular Float32Array silently
- Add `getSharedBuffers()` method that returns all property SharedArrayBuffer references (for transfer to worker)
- Add `isUsingSharedBuffers()` method that returns whether SharedArrayBuffer is in use
- All existing methods (getCurrentBuffer, getNextBuffer, swap, getCellValue, setCellValue) work identically regardless of backing store
- No changes to the existing API surface -- only additions
</task>

<task id="4">
Create `src/engine/grid/__tests__/grid-shared-buffer.test.ts`:
- TestGrid_SharedBuffer_CreatesWithSAB: create grid with `useSharedBuffer: true`, verify `isUsingSharedBuffers()` returns true (if SAB available in test env)
- TestGrid_SharedBuffer_FallsBackSilently: when SAB is unavailable, grid still works with regular ArrayBuffer
- TestGrid_SharedBuffer_BufferAccessIdentical: getCellValue/setCellValue work the same with shared buffers
- TestGrid_SharedBuffer_SwapWorksCorrectly: ping-pong swap works with shared buffers
- TestGrid_SharedBuffer_SameViewFromMultipleReferences: two Float32Array views on the same SharedArrayBuffer see the same data (simulates worker + main thread sharing)
- TestGrid_RegularBuffer_StillWorks: existing grid creation without `useSharedBuffer` continues to work unchanged

Note: SharedArrayBuffer may not be available in jsdom test environment. Use conditional tests that skip gracefully if SAB is undefined, and include at least one test that mocks SAB availability.
</task>
