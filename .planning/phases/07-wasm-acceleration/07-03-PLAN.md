# Plan 07-03: RuleRunner WASM Integration, Fallback, and Preset Updates

## Objective

Wire the Rust WASM module into the existing RuleRunner so that presets with `rule.type: 'wasm'` automatically use the Rust implementation while falling back to TypeScript silently when WASM is unavailable. Update preset schema to support 'wasm' rule type, update Gray-Scott and Navier-Stokes presets, and integrate WASM loading into the Web Worker.

## must_haves

- Preset schema accepts `rule.type: 'wasm'` alongside 'typescript'
- Gray-Scott and Navier-Stokes YAML presets updated to `rule.type: 'wasm'` with TypeScript compute body retained as fallback
- WasmRuleRunner class that loads WASM module and delegates tick() to Rust functions
- RuleRunner detects `rule.type: 'wasm'` and attempts WASM loading; falls back to TypeScript on failure
- Removing the .wasm file causes silent fallback -- no error thrown, simulation continues
- isUsingWasm() returns true when WASM is loaded, false when using TS fallback
- Worker loads WASM module inside the worker context
- WASM tick operates on Grid buffers directly (whole-tick API)
- Integration tests verify full WASM pipeline: preset load -> WASM init -> tick -> correct output
- Scenario test verifies fallback behavior: WASM unavailable -> TS continues working
- All existing tests continue to pass (no regression)

<task id="5">
Create `src/engine/rule/WasmRuleRunner.ts`:
- Class that wraps WASM module calls for a specific rule
- Constructor takes: Grid, PresetConfig, and loaded WASM module reference
- `tick()` method:
  - Gets current and next buffers for each property from Grid
  - For Gray-Scott: calls `wasmModule.gray_scott_tick(u_current, v_current, u_next, v_next, width, height, Du, Dv, F, k, dt)`
  - For Navier-Stokes: calls `wasmModule.navier_stokes_tick(...)` with all 4 property buffer pairs
  - Calls `grid.swap()` after the WASM function returns
  - Increments generation counter
- `isUsingWasm()` returns true
- Implements `IRuleRunner` interface
- Method `getWasmFnName(preset: PresetConfig): string | null` maps preset wasm_module field to the function name
</task>

<task id="6">
Update `src/engine/rule/RuleRunner.ts`:
- Replace `tryLoadWasm()` stub with actual WASM module loading:
  - If `preset.rule.type === 'wasm'`, attempt to dynamically import the wasm-bindgen generated module
  - Import path: try `import('../../wasm/pkg/lattice_engine.js')` (the generated JS glue)
  - If import succeeds and the expected function exists (e.g., `gray_scott_tick`), create a WasmRuleRunner delegate
  - If import fails (file missing, function not found, any error), log a warning and fall back to TypeScript silently
- When WASM delegate exists, `tick()` delegates to `wasmDelegate.tick()` instead of the per-cell JS loop
- `isUsingWasm()` returns `wasmDelegate !== null`
- For `rule.type: 'typescript'`, behavior is unchanged (always uses TS)
- The compute body for TS fallback comes from `preset.rule.compute` or `preset.rule.fallback_compute`
- Make tryLoadWasm async -- adjust constructor to support async initialization or use a static factory method `RuleRunner.create(grid, preset)`
</task>
